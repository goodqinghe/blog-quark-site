博客优化与供应链安全简介

写在前面

最近重构了一下这个博客的源码仓库，顺手把一些”以前没认真想过”的概念理了理。

这篇文章分两部分：

  • 第一部分：记录这次重构过程中我觉得有用的几条”博客优化方法”，基本都是朴素实践
  • 第二部分：顺带整理一下”供应链安全”这个之前听过但没仔细想过的概念，结合本仓库的具体实践

不是教程，只是备忘。

第一部分：简单的博客优化方法

1. 让部署流程自动化

博客最常见的”卡脖子点”是构建→推送→上线的链路。手动跑命令容易出错、容易忘。

最朴素的自动化是写一个本地脚本串起所有步骤。更好一点的是把流程交给 CI：

  • 推送到 main 即触发构建
  • 构建产物自动同步到发布仓库
  • 全程不需要本地操作

本博客的部署流程就属于这种——所有事情都在 .github/workflows/deploy.yml 里完成。

2. 让日志可读

如果自动化跑挂了，要能快速定位问题。这意味着：

  • CI 步骤名称要清晰，不要全是 “Run script 1”
  • 关键步骤之间加注释
  • 出错时不要静默吞掉，CI 失败要让通知发出来

3. 让”修改清单”可追溯

每次发布，应该能回答”这次发了什么”。最低成本的做法：

  • commit message 写清楚改了什么
  • 公开仓库的 commit 用统一前缀（如 build:）开头，log 才不会乱

4. 分离”源码”与”产物”

私有源码仓库 + 公开发布仓库这种分离方式有几个好处：

  • 源码可以公开也可以不公开（保护草稿、未发布内容）
  • 发布仓库只放构建产物，体积小、下载快
  • 公开仓库的 git log 只反映”什么时候发了什么”，不会泄露源码修改历史

5. 写 README 给未来的自己

文档写给”半年后回来想改点什么的自己”最有价值。内容包括：

  • 仓库结构图
  • 日常操作（怎么加文章、怎么发版）
  • 已知坑（”不要在 Y 之前 commit X”）

第二部分：供应链安全简介

什么是供应链攻击

软件供应链攻击指的是：攻击者不直接攻击你，而是攻击你依赖的东西——你用的库、你调用的 action、你下载的镜像、你引用的 CDN。

一旦被污染，你以为安全的代码实际上已经在执行恶意逻辑。

常见的攻击面

对个人博客/小项目来说，主要供应链攻击面是这几类：

攻击面例子
第三方 actionGitHub Actions 里 uses: some-user/some-action@main
第三方库package.json 里的 lodash 被植入后门
构建工具CI 里装的 Quarkdown、Node.js 被替换
镜像/二进制docker pull 拉到被篡改的镜像

防御的核心思路

没有银弹，但有几个朴素且有效的原则：

  1. 锁版本：不要写 @main / @latest，写具体的 tag 或 commit SHA
  2. 勤更新：定期 review 依赖的安全公告，但每次更新要可追溯
  3. 最小权限：CI 不要给过宽的 token，能用 deploy key 就不用 PAT
  4. 本地验证：重要依赖手动 review 一下 README、Star 数、Commit 历史
  5. 出问题时能停：依赖要能快速替换或临时移除

本博客的实践

回头看本博客的 deploy.yml，正好就是供应链安全原则的实战例子：

1. Action pin 到 commit SHA

uses: actions/checkout@9c091bb21b7c1c1d1991bb908d89e4e9dddfe3e0
# 而不是 @v4 或 @main

@main 的风险是：action 维护者随时可以 push 改动，你的 workflow 行为当天就会变。pin 到 commit SHA 意味着：除非这个具体 commit 被改动（不可能，因为它已经存在），否则行为不会变。

代价是要定期手动更新 SHA（每 3-6 个月看一次）。

2. 提交身份统一

user-email: "qinghe-dev@outlook.com"

CI 推送 commit 时用的 email 跟本地保持一致，避免公开仓库出现”两个作者”的奇怪对比，也方便日后排查。

3. commit message 加前缀

commit-message: |
  build: 同步博客构建产物
  来源仓库: ...
  源码 commit: ...
  源码 message: ...

公开仓库的 log 全是 build: 开头，干净统一；同时保留对源码 commit 的追溯链路。

4. files/ 目录的脱敏约定

所有放进 files/ 的代码片段都是对公网完全开放的。所以约定：

  • 没有真实凭据、token、SSH key
  • 涉及个人/公司的字段用 <示例值>、example.com 等占位符
  • 提交前按脱敏清单逐项检查

还想做但还没做的

  • 加 workflow_dispatch（手动触发）和 pull_request 触发器（PR 时 dry-run），让 CI 反馈更及时
  • workflow 加 actions/cache，减少重复下载依赖
  • 引入 Dependabot，自动监控依赖更新
